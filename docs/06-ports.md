# Port System

> Архитектура системы портов.

Version 0.1

---

# 1. Назначение

Порты являются единственной точкой подключения дорожек к компонентам.

Никакая дорожка не может подключаться непосредственно к компоненту.

Все соединения проходят через порт.

---

# 2. Главный принцип

Соединение всегда выглядит следующим образом.

```text
Track

↓

Connection

↓

Port

↓

Component
```

Это фундаментальное правило LogicCAD.

---

# 3. Почему существуют порты

Порты позволяют:

* явно определить точки подключения;
* выполнять проверку схем;
* выполнять компиляцию;
* реализовать подсветку;
* отделить компонент от дорожек.

---

# 4. Definition

Каждый тип компонента определяет собственный набор портов.

Например

Arithmetic Definition

имеет

```text
Green Input

Green Output

Red Input

Red Output
```

Редактор не создаёт порты самостоятельно.

---

# 5. Instance

Экземпляры компонентов не содержат экземпляров портов.

Положение порта вычисляется.

```text
Definition

+

Rotation

+

Position

↓

Screen Position
```

---

# 6. Геометрия

Каждый порт имеет локальную координату.

Например

```text
x = 0

y = -1
```

После поворота координата автоматически пересчитывается.

---

# 7. Типы портов

В MVP существуют четыре типа.

```text
Green Input

Green Output

Red Input

Red Output
```

Архитектура допускает расширение.

---

# 8. Каналы

Каждый порт предоставляет один логический канал.

В текущей версии доступны:

* Red
* Green

В будущем возможно появление других каналов без изменения общей модели.

---

# 9. Connection

Connection является отдельной сущностью.

Она соединяет дорожку и порт.

```text
Track

↓

Connection

↓

Port
```

Порт не хранит список подключённых дорожек.

---

# 10. Snap

Во время рисования дорожки выполняется привязка.

Пользователь может завершить дорожку только:

* на порту;
* на другой дорожке.

Это исключает недопустимые соединения.

---

# 11. Ограничения

Нельзя подключить дорожку:

* к корпусу компонента;
* в произвольную точку рядом;
* к пустому месту.

---

# 12. Validation

При проверке документа контролируется:

* существование порта;
* существование компонента;
* допустимость соединения;
* корректность координат.

---

# 13. Подсветка

При выборе порта подсвечиваются все дорожки соответствующей сети.

Если дорожка относится сразу к нескольким сетям, отображается комбинированная подсветка.

---

# 14. Rotation

После поворота компонента:

* координаты портов пересчитываются;
* существующие Connection остаются привязанными к тем же портам;
* если геометрия становится недопустимой, Validation сообщает об ошибке.

---

# 15. Runtime

Во время компиляции порты используются только для определения того, какие сущности необходимо соединить проводами.

В Runtime отдельной сущности Port не существует.

---

# 16. Port API

Каждый порт предоставляет единый интерфейс.

Например:

* получить координаты;
* получить тип;
* получить компонент-владелец;
* получить направление;
* определить доступные каналы.

---

# 17. Независимость

Port ничего не знает:

* о дорожках;
* о Runtime;
* о GUI.

Port описывает только точку подключения.

---

# 18. Будущие возможности

Архитектура должна позволять добавить:

* многоканальные порты;
* пользовательские порты;
* шины;
* тестовые точки;
* внутренние соединители;
* соединение между вложенными компонентами.

Без изменения существующей модели.

---

# 19. Основные принципы

Порт:

* принадлежит Definition;
* вычисляется динамически;
* используется Editor;
* используется Compiler;
* не сериализуется как отдельный объект экземпляра.
