# Component System

> Архитектура компонентов LogicCAD.

Version 0.1

---

# 1. Назначение

Компоненты являются основными строительными блоками схем.

Каждый компонент представляет некоторую логическую функцию.

Например:

* Arithmetic Combinator
* Decider Combinator
* Constant Combinator

Редактор никогда не работает напрямую с объектами Factorio.

Он работает только с компонентами.

---

# 2. Главная идея

Компонент состоит из двух частей.

```
Component Definition

↓

Component Instance
```

Это одно из ключевых архитектурных решений проекта.

---

# 3. Component Definition

Definition описывает тип компонента.

Он существует один раз.

```
Arithmetic Definition
```

Все арифметические комбинаторы используют одну Definition.

---

Definition содержит:

* имя
* размер
* иконку
* описание портов
* параметры
* правила компиляции
* правила отображения

---

# 4. Component Instance

Instance представляет размещённый компонент.

```
Arithmetic #14
```

Instance содержит только:

* id
* definition
* координаты
* поворот
* пользовательские настройки

---

# 5. Архитектура

```
Document

↓

Component Instance

↓

Component Definition
```

Definition никогда не хранится внутри документа.

---

# 6. Registry

Все определения компонентов регистрируются.

```
Component Registry

↓

Arithmetic

↓

Decider

↓

Constant
```

Registry является единственным местом регистрации компонентов.

---

# 7. Definition

Каждая Definition должна содержать:

```
Id

Display Name

Size

Ports

Configuration

Editor

Compiler
```

---

# 8. Instance

Экземпляр содержит только изменяемые данные.

```
ComponentId

DefinitionId

Position

Rotation

Configuration
```

---

# 9. Position

Положение хранится исключительно в координатах редактора.

```
x

y
```

Используются только целые значения.

---

# 10. Rotation

Поддерживаются четыре направления.

```
North

East

South

West
```

Поворот не изменяет Definition.

---

# 11. Configuration

Configuration содержит настройки пользователя.

Например для Arithmetic:

```
Operation

Left Signal

Right Signal

Output Signal
```

Configuration принадлежит Instance.

---

# 12. Порты

Definition определяет расположение портов.

Например

```
Green Input

Green Output

Red Input

Red Output
```

Экземпляр компонента не хранит собственные порты.

---

# 13. Port Layout

После поворота компонента положение портов вычисляется.

```
Definition

+

Rotation

↓

Screen Position
```

Это избавляет от хранения лишних данных.

---

# 14. Размер

Каждый компонент имеет размер.

Например

```
1×1

2×1

2×2
```

Размер определяется Definition.

---

# 15. Размещение

При размещении выполняются проверки.

* свободное место
* отсутствие пересечения
* допустимость координат

После успешной проверки создаётся Instance.

---

# 16. Component API

Все компоненты используют единый интерфейс.

Например:

```
Create()

Rotate()

Validate()

Compile()

Render()
```

Редактор никогда не проверяет тип компонента вручную.

---

# 17. Rendering

Definition предоставляет информацию для визуализации.

Например:

* спрайт
* размеры
* положение портов
* область выделения

---

# 18. Compiler

Каждый тип компонента самостоятельно знает, как создать Runtime.

Например

```
Arithmetic

↓

Runtime Entity
```

Компилятор вызывает Definition.

---

# 19. Validation

Definition также отвечает за собственную проверку.

Например

Arithmetic проверяет:

* выбранную операцию;
* корректность сигналов;
* корректность выхода.

Редактор не знает внутреннего устройства компонентов.

---

# 20. Registry API

Регистрация нового компонента выглядит примерно так:

```
register_component(
    definition
)
```

После регистрации компонент автоматически становится доступен:

* редактору;
* компилятору;
* GUI;
* Runtime Builder.

---

# 21. Стандартные компоненты MVP

В первой версии доступны:

* Arithmetic
* Decider
* Constant

Они поставляются встроенными.

---

# 22. Пользовательские компоненты

Архитектура должна позволять позже добавить:

* собственные компоненты;
* библиотеки компонентов;
* импорт компонентов;
* экспорт компонентов.

Без изменения существующего кода.

---

# 23. Требования

Definition должна быть:

* неизменяемой;
* независимой от документа;
* независимой от Runtime;
* независимой от Factorio API.

Instance должна содержать только пользовательские данные.

---

# 24. Принцип

Definition описывает,

Instance хранит,

Editor изменяет,

Compiler использует,

Runtime исполняет.

Каждая подсистема имеет только одну ответственность.
