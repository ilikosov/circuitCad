# Vision

> Архитектурное видение проекта LogicCAD.

Version: 0.1 (Draft)

---

# 1. Назначение документа

Этот документ описывает долгосрочное видение проекта.

Он не содержит технических деталей реализации.

Его задача — объяснить:

- зачем существует LogicCAD;
- какие проблемы он решает;
- какие архитектурные принципы нельзя нарушать;
- каким должен стать проект через несколько лет.

Любое архитектурное решение должно соответствовать данному документу.

---

# 2. Проблема

Circuit Network в Factorio является чрезвычайно мощной системой.

Однако по мере роста проекта возникают серьезные проблемы.

Например:

- десятки арифметических комбинаторов;
- сотни соединений;
- длинные цепочки проводов;
- невозможность быстро определить принадлежность сигнала;
- сложность поиска ошибок.

Через некоторое время схема становится практически нечитаемой.

Проблема заключается не в возможностях Circuit Network.

Проблема заключается в отсутствии инструментов проектирования.

---

# 3. Что такое LogicCAD

LogicCAD — это не замена Circuit Network.

LogicCAD — это инструмент разработки.

Он занимает такое же место, как:

- KiCad для печатных плат;
- Logisim для цифровой логики;
- Visual Studio для программирования.

После завершения работы пользователь получает обычную схему Factorio.

---

# 4. Основная идея

Вместо работы с проводами пользователь работает с логической моделью.

```
             Пользователь

                  │

             LogicCAD Editor

                  │

              Компилятор

                  │

          Circuit Network Factorio
```

LogicCAD отвечает только за:

- проектирование;
- хранение;
- компиляцию.

Выполнение остается полностью за движком игры.

---

# 5. Главный принцип

## Никогда не реализовывать собственный логический движок.

Это важнейшее архитектурное решение проекта.

Причины:

- гарантированная совместимость;
- отсутствие различий между редактором и игрой;
- минимальная вероятность ошибок;
- поддержка любых изменений Circuit Network после обновлений Factorio.

LogicCAD не вычисляет сигналы.

LogicCAD только строит схему.

---

# 6. Editor First

Единственным источником истины является редактор.

```
Document

↓

Compile

↓

Runtime
```

Runtime никогда не редактируется.

Если Runtime поврежден,

он полностью удаляется и создается заново.

---

# 7. Runtime Disposable

Runtime считается временным объектом.

Его всегда можно восстановить.

Это означает:

- отсутствие синхронизации;
- отсутствие ручных изменений;
- отсутствие миграций Runtime.

Все миграции происходят только на уровне Document.

---

# 8. WYSIWYG

Редактор определяет:

- расположение компонентов;
- соединения;
- ориентацию.

Компилятор не имеет права:

- переставлять компоненты;
- менять соединения;
- менять топологию.

```
Editor

=

Runtime
```

---

# 9. CAD вместо Sandbox

Factorio позволяет свободно строить схемы.

LogicCAD вводит ограничения.

Почему?

Потому что ограничения позволяют:

- автоматически проверять ошибки;
- анализировать схему;
- строить Net Graph;
- выполнять компиляцию.

Свободное размещение проводов заменяется дорожками.

---

# 10. Явные порты

В LogicCAD соединяются не сущности.

Соединяются порты.

```
Component

○ Green In

○ Green Out

○ Red In

○ Red Out
```

Это делает схему более читаемой.

---

# 11. Дорожки

Провода заменяются дорожками.

Причины:

- лучшая читаемость;
- регулярная сетка;
- автоматическая подсветка;
- простая трассировка.

Дорожки не имеют направления.

Они определяют только принадлежность сети.

---

# 12. Net Graph

Во время компиляции дорожки преобразуются в граф.

```
Tracks

↓

Net Graph

↓

Compiler
```

Компилятор не работает с дорожками.

Компилятор работает только с графом.

---

# 13. Компоненты

Компонент редактора —

это не LuaEntity.

Это независимая модель.

Например:

```
Arithmetic

Position

Rotation

Operation

Signals
```

Компонент можно:

- сериализовать;
- копировать;
- тестировать;
- валидировать.

---

# 14. Документ

Редактор работает не с поверхностью.

Редактор работает с документом.

```
Document

Components

Tracks

Metadata
```

Поверхность является только визуальным отображением документа.

---

# 15. Компилятор

Компилятор должен быть максимально простым.

Он не принимает решений.

Он только переносит Document в Runtime.

Это делает поведение полностью предсказуемым.

---

# 16. Простота важнее оптимизации

На первом этапе предпочтение отдается:

- понятному коду;
- детерминированному поведению;
- полной перекомпиляции.

Оптимизация возможна позже.

---

# 17. Расширяемость

Архитектура должна позволять добавление:

- новых компонентов;
- пользовательских библиотек;
- иерархических компонентов;
- шаблонов;
- новых типов дорожек;
- новых режимов компиляции.

Без изменения существующего кода.

---

# 18. Совместимость

Внешний компонент должен выглядеть как обычный комбинатор.

Игрок должен иметь возможность подключить его к:

- Arithmetic Combinator;
- Decider Combinator;
- Constant Combinator;
- Lamp;
- Programmable Speaker;
- любому совместимому моду.

---

# 19. Долгосрочная цель

Конечной целью проекта является создание полноценной CAD-системы для Circuit Network.

Пользователь должен иметь возможность проектировать схемы любого масштаба без потери читаемости.

Factorio становится вычислительным движком.

LogicCAD становится инженерным инструментом.
